# LightRAG Developer Guide

A practical walkthrough for running and testing the LightRAG-backed knowledge base in `ntg-agent`. Companion to [`CLAUDE.md`](./CLAUDE.md) — that file is architectural; this one is operational.

## What LightRAG does here

- Acts as the **document RAG backend** (replaced Kernel Memory for document RAG; KM still owns long-term user memory).
- **One LightRAG container per agent.** The AppHost starts only the shared storage:
  - `lightrag-postgres` — custom PG17 image with `pgvector` + Apache AGE (`scripts/lightrag-postgres/Dockerfile`). Hosts the `uploaded-documents` database. This is the single durable backend for **all** agents.
  - There is **no** singleton `lightrag` container anymore. Instead the Orchestrator spins up one lightweight app container per agent — `lightrag-agent-{agentId}` (upstream `ghcr.io/hkuds/lightrag:v1.4.16`) — on the host Docker daemon via `Docker.DotNet`. Each joins the same Docker network as `lightrag-postgres`, talks to Azure OpenAI for extraction/embeddings, and is isolated by LightRAG's `WORKSPACE={agentId}` env var so two agents never see each other's data even though they share one Postgres.
- **Isolation model:** a `lightrag` container is a stateless app server; all state (KV/vector/doc-status/AGE graph) lives in Postgres, where every row is namespaced by `WORKSPACE`. Distinct workspaces ⇒ disjoint rows ⇒ full per-agent isolation without a Postgres-per-agent.
- **Container lifecycle is orchestrator-owned** (`Services/Knowledge/`):
  - `LightRagContainerManager` — pull image, find a free host port, create/start/stop/remove `lightrag-agent-{id}`.
  - `LightRagClientFactory` — resolves a `LightRagClient` pointed at the agent's own container (`http://localhost:{Agent.LightRagPort}`); this is how chat/upload stay scoped to one agent.
  - `LightRagReconcilerHostedService` — on startup, pulls the image and ensures every agent has a running container, back-filling/repairing `Agent.LightRagPort`.
  - Create-agent spawns the container and persists its port; delete-agent cascade-deletes the agent's documents (LightRAG → filestore → SQL) then tears the container down (`AgentAdminController`).
- The dynamically-allocated host port for each agent's container is persisted on the `Agents.LightRagPort` column in SQL Server (the only per-agent metadata we store).
- Persists chunks/embeddings/graph in the `ntg-agent-local-dev-lightrag-postgres-data` Docker volume.
- Stores original uploaded files locally at `./NTG.Agent.Orchestrator/lightrag-filestore/{agentId}/{docId}_{fileName}` (slated to move to Azure Blob in prod).
- Each agent container exposes the built-in WebUI at <http://localhost:{Agent.LightRagPort}/webui/> — invaluable for debugging ingestion state per agent.

## Prerequisites

1. **.NET 10 SDK**, **Docker** (running), and `dotnet-ef` installed globally (`dotnet tool install --global dotnet-ef`).
2. **AppHost user-secrets**. The LightRAG-relevant ones:
   - `lightrag-pg-password` — auto-generated by the secrets script if absent.
   - `lightrag-api-key` — gates the LightRAG HTTP API; auto-generated if absent.
   - `azure-openai-api-key` — the Azure key for the resource hosting `gpt-5.4-mini` (entity extraction).
   - `azure-embedding-api-key` — the Azure key for the resource hosting `text-embedding-3-large` (embeddings).

   Run `./scripts/init-apphost-user-secrets.sh` and follow the prompts (see `CLAUDE.md` for the full secrets list).

## Required Azure OpenAI deployments

LightRAG expects these **deployment names** to exist on the two Azure resources (Azure OpenAI deployment names are arbitrary — if your portal uses different names, update `LLM_MODEL` / `EMBEDDING_MODEL` in `NTG.Agent.AppHost/Program.cs`):

| Endpoint | Deployment name | Used by |
| --- | --- | --- |
| `https://rmit-capstone-2026-resource.cognitiveservices.azure.com/` | `gpt-5.4-mini` | LightRAG entity extraction + KM text gen |
| `https://rmit-capstone-2026-ext-resource.cognitiveservices.azure.com/` | `text-embedding-3-large` | LightRAG embeddings + KM embeddings |

## First-run setup

```bash
# From repo root
dotnet restore
dotnet build --configuration Release

# Interactive: prompts for each secret, reuses .env if present
./scripts/init-apphost-user-secrets.sh

# Boots SQL Server, Postgres+pgvector+AGE, LightRAG, Elasticsearch+Kibana,
# runs EF migrations, then starts every .NET service
dotnet run --project NTG.Agent.AppHost
```

Wait for `lightrag-postgres`, `lightrag`, and `ntg-agent-orchestrator` to reach **Running** in the Aspire dashboard. First-time `lightrag-postgres` boot compiles Apache AGE from source — expect ~3–5 min.

## Verifying LightRAG is healthy

1. **WebUI**: open <http://localhost:9621/webui/>. The Documents tab is empty on a fresh volume.
2. **API ping**:
   ```bash
   LIGHTRAG_KEY=$(dotnet user-secrets list --project NTG.Agent.AppHost/NTG.Agent.AppHost.csproj | awk -F'= ' '/lightrag-api-key/ {print $2}')
   curl -s -H "X-API-Key: $LIGHTRAG_KEY" http://localhost:9621/health
   ```
   Expect HTTP 200.
3. **Aspire logs**: the `lightrag` container should NOT log `401 Unauthorized` from Azure. If it does, your `azure-openai-api-key` or `azure-embedding-api-key` is wrong or pointed at the wrong resource.

## Uploading a document end-to-end

### Via the Admin Blazor UI (the supported path)

1. Open the admin URL from the Aspire dashboard (`ntg-agent-admin`). Login with `admin@ntgagent.com` / `Ntg@123`.
2. **Agents → Default Agent → Knowledge Base tab → Add Knowledge**. Pick a PDF/docx/xlsx/txt and submit.
3. Under the hood (non-blocking, status-aware): `POST /api/Documents/upload/{agentId}` (Orchestrator) → `LightRagKnowledge.BeginImportDocumentAsync` saves the original file to `./NTG.Agent.Orchestrator/lightrag-filestore/{agentId}/{documentId}_{fileName}` (keyed by the local `Document.Id`, so download works regardless of ingestion state) and calls `LightRagClient.InsertFileAsync` → LightRAG `/documents/upload` (async queue), which returns a `track_id` immediately. The orchestrator commits a `Document` row right away with `Status = Processing` and that `TrackId`, then returns — it does **not** wait for extraction. (A *begin* failure, e.g. the container being unreachable, returns a per-file failure and commits no row.)
4. A background worker (`LightRagIngestionStatusHostedService`) advances the row: it polls `/documents/track_status/{track_id}` via `LightRagKnowledge.CheckIngestStatusAsync` every `LightRag:PollIntervalSeconds` (default 3s) and flips the row to `Completed` (filling `KnowledgeDocId` with the real `doc-XXX`) or `Failed` (filling `ErrorMessage`). The Admin Knowledge Base list shows the live status as **Uploading → Uploaded / Failed to upload** and auto-refreshes while anything is still "Uploading". The worker is **event-driven**: it stays parked (no polling, no DB queries) until an upload wakes it via `IngestionStatusSignal.Notify()`, drains every `Processing` doc, then parks again. A drain pass also runs on startup to recover docs left `Processing` by a previous run. Watch the LightRAG WebUI Documents tab for a live view of the pipeline stages (`Pending → Processing → Completed/Failed`).

### Via curl (when you want to bypass auth/UI for testing)

```bash
curl -H "X-API-Key: $LIGHTRAG_KEY" \
     -F "file=@./my-doc.pdf" \
     http://localhost:9621/documents/upload
```

This skips the Orchestrator entirely — the doc lands in LightRAG but **no `Document` row is created in `AgentDbContext`**, so the Admin UI Knowledge Base tab won't see it. Use only for isolating LightRAG-side bugs.

### Verifying it landed

- WebUI Documents tab → row with `Completed` status, non-zero `Chunks` column.
- API: `curl -s -H "X-API-Key: $LIGHTRAG_KEY" http://localhost:9621/documents | jq` — should include the new `doc-XXX` ID.

## Verifying chat-side retrieval works

Once at least one document is `Completed` in LightRAG:

1. **Confirm the Default Agent is on `AzureOpenAI`**: Admin → Agents → Default Agent → Agent Settings. The Provider Name field must be exactly `AzureOpenAI` (case-sensitive — see `AgentFactory.cs:37-45`). If it's still `GitHubModel`, set it now:
   - Provider Name: `AzureOpenAI`
   - Provider Endpoint: `https://rmit-capstone-2026-resource.cognitiveservices.azure.com/`
   - Provider Model Name: `gpt-5.4` (deployment name)
   - Provider API Key: same value as your `azure-openai-api-key` user-secret.

2. Open the WebClient, start a chat, and ask a question that should pull from your uploaded doc.

3. In the Orchestrator logs (Aspire → `ntg-agent-orchestrator`), look for:
   - `LightRagClient.QueryAsync` followed by `HTTP POST http://localhost:9621/query` returning 200.
   - The chat response should include grounded content from your document.

## Common failure modes

### `HTTP 413 tokens_limit_reached` during chat

- **Cause**: the chat agent's `ProviderName` is still `GitHubModel`/`OpenAI` and `gpt-4.1`, which caps requests at 8000 tokens. LightRAG's `memory` plugin can return ~12K tokens of context — request blows the cap.
- **Fix**: change the agent's Provider to `AzureOpenAI` (see "Verifying chat-side retrieval" above). No restart needed — `AgentFactory.CreateAgent` re-reads the DB row on every chat call.

### `HTTP 413 tokens_limit_reached` during LightRAG ingestion

- **Cause**: LightRAG container was started against GitHub Models (old config).
- **Fix**: verify `NTG.Agent.AppHost/Program.cs:55-69` sets `LLM_BINDING=azure_openai` and your Azure user-secrets are populated. Restart the AppHost so the new env vars apply.

### `429 Too many requests` during ingestion

- **Cause**: same as above — fell back to GitHub Models, which is heavily rate-limited.
- **Fix**: verify Azure OpenAI binding. If you must use GitHub Models temporarily, also set `MAX_ASYNC=1` and `MAX_PARALLEL_INSERT=1` in the LightRAG container env.

### `TaskCanceledException` 100s timeout in chat (`UserMemoryService` stack)

- **Cause**: Kernel Memory (long-term user memory store, distinct from LightRAG) is on GitHub Models embeddings and hangs.
- **Fix**: verify `NTG.Agent.Knowledge/appsettings.Development.json` has `TextGeneratorType=AzureOpenAIText` and `EmbeddingGeneratorType=AzureOpenAIEmbedding`. If you flipped this after KM had written to Elasticsearch, wipe the ES volume (see next item).

### Embedding-dimension mismatch (silent breakage)

- **Symptom**: KM `/search` returns 0 results, or LightRAG ingestion errors mention pgvector dimensions.
- **Cause**: changed the embedding model after data was already ingested — pgvector / Elasticsearch indexes are locked to the dimension of whatever embedded the first row.
- **Fix**: stop the AppHost, wipe the affected volume, restart:
  ```bash
  docker volume rm ntg-agent-local-dev-lightrag-postgres-data   # for LightRAG
  docker volume rm ntg-agent-local-dev-elasticsearch-data       # for KM
  ```

### A document is stuck at "Uploading"

- **Expected**: ingestion is asynchronous. The upload call returns immediately with the row in `Processing`; the background worker (`LightRagIngestionStatusHostedService`) flips it to `Uploaded`/`Failed` once LightRAG finishes. Large docs can take 1–2 min to leave "Uploading" — the Admin list auto-refreshes.
- **If it never leaves "Uploading"**: check the agent's LightRAG WebUI for the pipeline stage (`Pending → Processing → Completed/Failed`), and the Orchestrator logs for `Failed to check ingestion status` warnings (e.g. the agent's container is down). The worker retries every tick and only stops polling once no doc is `Processing`.
- **Tuning**: `LightRag:PollIntervalSeconds` (default 3) sets the worker's poll cadence. (`LightRag:UploadTimeoutSeconds` is now **unused** — the upload request no longer blocks on extraction, so there is no per-request timeout to tune.)

### `database "uploaded-documents" does not exist` or `extension "age" is not available`

- **Cause**: the `lightrag-postgres` data volume was created before the init scripts changed (init scripts only run on a fresh volume).
- **Fix**: `docker volume rm ntg-agent-local-dev-lightrag-postgres-data`, then restart the AppHost so init scripts run on the empty volume.

## Tunable knobs (current values)

Set as `LightRag__*` env vars on the Orchestrator in `NTG.Agent.AppHost/Program.cs` (bound to `LightRagSettings`) and applied by `LightRagContainerManager` to every per-agent container:

| Env var | Current | LightRAG default | What it controls |
| --- | --- | --- | --- |
| `CHUNK_SIZE` | 1500 | 1200 | Tokens per chunk during partitioning. Bigger = fewer LLM calls, more context per call. |
| `CHUNK_OVERLAP_SIZE` | 100 | 100 | Token overlap between adjacent chunks. |
| `MAX_ASYNC` | 8 | 4 | Max concurrent LLM calls. Azure OpenAI gpt-5.4-mini can handle this; drop to 1 if you ever revert to GitHub Models. |
| `MAX_PARALLEL_INSERT` | 2 | 2 | Documents processed in parallel by the pipeline. |
| `EMBEDDING_FUNC_MAX_ASYNC` | 8 | 8 | Max concurrent embedding calls. |

LightRAG settings on the Orchestrator side (in `appsettings.json`, section `LightRag`):

| Setting | Default | What it controls |
| --- | --- | --- |
| `TopK` | 60 | Number of entities/relations retrieved from the KG per query. |
| `PollIntervalSeconds` | 3 | How often `LightRagIngestionStatusHostedService` polls LightRAG for `Processing` docs while draining. |
| `UploadTimeoutSeconds` | 180 | **Unused** — legacy of the old synchronous upload model; ingestion no longer blocks the request. |

`LightRagClient.QueryAsync` is hard-coded to request up to ~12K tokens of context per query (4000 each for text-unit / global / local). Tune in `LightRagClient.cs:52` if you need smaller responses (e.g. when paired with a small-context chat model).

## Known limitations

- **One container per agent costs RAM**: each agent runs its own LightRAG app container. Fine for a handful of agents in dev; a large fleet would want a pooled/multi-tenant topology or on-demand container start/stop. Containers use `RestartPolicy=unless-stopped` and are reconciled on Orchestrator startup.
- **Local filestore**: original uploads still live on the Orchestrator's disk (`lightrag-filestore/{agentId}/`) — move to Azure Blob for production.
- **Pre-existing orphans**: the async model commits the `Document` row before ingestion, so it won't create new orphans. But documents ingested directly into LightRAG (e.g. via curl, or by an older build that timed out) have no `Document` row and won't appear in the Admin UI — clean those up via the LightRAG WebUI.
