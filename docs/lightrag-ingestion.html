LightRAG ingestion — how uploaded items are processed

How an uploaded item (file, pasted text, or web page) travels from the HTTP request to a queryable document in its agent's LightRAG workspace. The design is non-blocking: the upload request returns as soon as LightRAG accepts the item and hands back a track_id; a background worker then advances the document to Completed or Failed.

Container provisioning (the per-agent LightRAG container reached over the SSH tunnel) is covered separately in LightRAG over an SSH tunnel — architecture; this document assumes LightRagClientFactory.GetClientAsync yields a client already pointed at the agent's running container.

1. The two phases at a glance

flowchart LR
    subgraph sync["Phase 1 — begin (synchronous, per upload request)"]
        u["client uploads item"] --> ctl["DocumentsController"]
        ctl --> begin["IKnowledgeService.BeginImport*"]
        begin --> lr1["POST to LightRAG → track_id"]
        begin --> fs["save original bytes to file store"]
        ctl --> row["Document row: Status = Processing, TrackId"]
        row --> notify["IngestionStatusSignal.Notify()"]
        ctl --> resp["200 OK returned immediately"]
    end
    subgraph async["Phase 2 — track (background, event-driven)"]
        notify -.wakes.-> worker["LightRagIngestionStatusHostedService"]
        worker --> poll["GET /documents/track_status/{trackId}"]
        poll --> apply["update Document → Completed / Failed"]
    end

Why two phases. LightRAG's /documents/upload and /documents/text are asynchronous — they return a track_id at once and assign the real doc-id later while the extraction pipeline runs. Blocking the HTTP request on that pipeline (chunking, embeddings, entity/relation extraction) would tie up the request for 1–2 minutes per large document, so the Orchestrator records the item as Processing and polls for the outcome out of band.

2. Phase 1 — the upload request (file)

Entry point: DocumentsController.UploadDocuments → LightRagKnowledge.BeginImportDocumentAsync → LightRagClient.InsertFileAsync.

sequenceDiagram
    autonumber
    participant U as Admin client
    participant Ctl as DocumentsController
    participant K as LightRagKnowledge
    participant CF as LightRagClientFactory
    participant FS as LightRagFileStore
    participant LR as LightRAG container (/documents/upload)
    participant DB as AgentDbContext
    participant Sig as IngestionStatusSignal

    U->>Ctl: POST /api/documents/upload/{agentId} (files)
    loop for each file (length > 0)
        Ctl->>K: BeginImportDocumentAsync(stream, name, agentId, documentId)
        K->>CF: GetClientAsync(agentId)
        CF-->>K: client for the agent's container
        K->>K: buffer stream (read twice)
        K->>FS: SaveAsync(agentId, documentId, name, bytes)
        Note over FS: original bytes downloadable while extraction runs
        K->>LR: POST /documents/upload (multipart)
        LR-->>K: { track_id, status } (async accept)
        K-->>Ctl: track_id
        Ctl->>Ctl: build Document row (Status = Processing, TrackId)
    end
    Ctl->>DB: AddRange(documents, tags) + SaveChangesAsync
    Ctl->>Sig: Notify() — wake the status worker
    Ctl-->>U: 200 OK [ per-file results ]

The BeginImport* call is wrapped per file in a try/catch: if LightRAG is unreachable at accept time, that file gets no row and is reported Success = false in the response, while the others still proceed. The Document row is only written once a track_id comes back.

3. Phase 2 — background status tracking

LightRagIngestionStatusHostedService is event-driven: it parks on the signal (no timer, no DB queries) while nothing is ingesting, wakes on Notify(), then drains — polling every PollIntervalSeconds (default 3s) while any document is still Processing.

sequenceDiagram
    autonumber
    participant Sig as IngestionStatusSignal
    participant W as IngestionStatusHostedService
    participant Store as ILightRagIngestionStore (EF)
    participant K as LightRagKnowledge
    participant LR as LightRAG container (/documents/track_status)

    Note over W: startup drain pass also runs once (recovers leftovers)
    W->>Sig: WaitAsync() — parked, no polling
    Sig-->>W: woken by an upload's Notify()
    loop while any document is Processing
        W->>Store: GetProcessingDocumentsAsync()
        Store-->>W: [ (documentId, agentId, trackId) ]
        loop each pending document
            W->>K: CheckIngestStatusAsync(agentId, trackId)
            K->>LR: GET /documents/track_status/{trackId}
            LR-->>K: { documents: [ { id, status, error_msg } ] }
            alt status = PROCESSED
                K-->>W: Completed (knowledgeDocId)
            else status = FAILED
                K-->>W: Failed (errorMessage)
            else PENDING / PROCESSING / PREPROCESSED / none
                K-->>W: Processing (keep for next pass)
            end
        end
        W->>Store: ApplyUpdatesAsync(Completed / Failed)
        Note over W: wait PollIntervalSeconds before the next pass
    end
    W->>Sig: WaitAsync() — park again

A per-document failure to reach its container is caught and counted as "still processing", so one agent's container being down never stalls the others — it is simply retried on the next pass. Updates are written back to the Document rows in a single batch via ApplyUpdatesAsync: Status, KnowledgeDocId, and ErrorMessage.

4. Inside the LightRAG container

What the container does between accepting the upload and reporting PROCESSED. All storage is the shared Postgres, scoped by the agent's WORKSPACE (KV, vector, graph, and doc-status backends).

sequenceDiagram
    autonumber
    participant LR as LightRAG API
    participant P as extraction pipeline
    participant AOAI as Azure OpenAI
    participant PG as Postgres (workspace-scoped)

    LR->>PG: record doc-status = PENDING (track_id)
    LR->>P: enqueue (returns track_id to caller)
    P->>P: split into chunks (CHUNK_SIZE / CHUNK_OVERLAP)
    P->>AOAI: embed chunks (text-embedding-3-large, dim = EMBEDDING_DIM)
    AOAI-->>P: vectors
    P->>AOAI: extract entities + relations (LLM)
    AOAI-->>P: graph elements
    P->>PG: upsert chunks + vectors (PGVectorStorage)
    P->>PG: upsert entities + relations (PGGraphStorage)
    P->>PG: upsert KV + doc-status = PROCESSED (assign doc-id)

The status the Orchestrator polls (track_status) mirrors this doc-status: PENDING → PROCESSING → PROCESSED on success, or FAILED with an error_msg. Tuning knobs (CHUNK_SIZE, CHUNK_OVERLAP_SIZE, MAX_ASYNC, MAX_PARALLEL_INSERT, EMBEDDING_FUNC_MAX_ASYNC) are set as container env by the container manager.

5. Document status lifecycle (Orchestrator side)

stateDiagram-v2
    [*] --> Processing: row written after track_id returned
    Processing --> Processing: track_status = PENDING / PROCESSING / PREPROCESSED
    Processing --> Completed: track_status = PROCESSED (KnowledgeDocId set)
    Processing --> Failed: track_status = FAILED (ErrorMessage set)
    Completed --> [*]: queryable via SearchAsync (hybrid)
    Failed --> [*]: surfaced in the documents list
    note right of Processing
        begin-step failure (backend
        unreachable) → no row at all,
        reported in the upload response
    end note

Delete while processing. Removing a document that never finished (only a track_id, no KnowledgeDocId yet) resolves the doc-id from the track_id first, so /documents/delete_document can clean up the item once extraction completes — no orphan left in LightRAG. The on-disk file is always removed from the file store.

6. Key components

ComponentRole in ingestion
DocumentsControllerAccepts uploads, writes the Processing row, calls Notify()
LightRagKnowledge (IKnowledgeService)Begin-import per item type; maps track_status → DocumentIngestResult; search / delete / export
LightRagClientTyped HTTP calls to the container: /documents/upload, /documents/text, /documents/track_status, /documents/delete_document, /query
LightRagFileStorePersists the original uploaded bytes (keyed by documentId) for later download
IngestionStatusSignalCoalescing wake signal (semaphore, max 1) that lets the worker park while idle
LightRagIngestionStatusHostedServiceBackground worker: drains Processing docs, applies Completed / Failed
ILightRagIngestionStore (EF)Reads Processing docs and writes status back to Document rows
