LightRAG over an SSH tunnel — architecture

The SSH-tunnel connectivity option for relocating LightRAG and its Postgres to the Azure Ubuntu VM, for the case where we cannot open an Azure NSG port but already have working SSH access.

The Orchestrator keeps driving the remote Docker daemon (Option 2), but all three runtime channels ride over one persistent SSH connection to the VM's already-open port 22 — no new inbound ports, no manager/Azure dependency.

ChannelSSH mechanismWhy
Docker daemon-L localhost:2375 → /var/run/docker.sockDrive the remote daemon; socket never exposed on TCP
Postgres (reset path)-L localhost:5432 → 127.0.0.1:5432ResetVectorSchemaAsync connects directly
Dynamic container HTTP ports-D 1080 (SOCKS5)Reach any runtime-assigned lightrag-agent port with no per-port setup

One command opens all three:

ssh -D 1080 \
    -L 2375:/var/run/docker.sock \
    -L 5432:127.0.0.1:5432 \
    ntgagent@4.193.109.6

The agent containers reach Postgres internally over the Docker network alias lightrag-postgres — that traffic never touches the tunnel. Container ports and Postgres bind to the VM's loopback; the Docker daemon stays on its unix socket (never on TCP).

3. RAG query (reaching a dynamic container port via SOCKS)

sequenceDiagram
    autonumber
    participant O as Orchestrator
    participant S as SOCKS proxy localhost:1080
    participant T as ssh -D tunnel
    participant C as lightrag-agent 127.0.0.1:P
    participant PG as lightrag-postgres
    O->>S: GET http://127.0.0.1:P/query (proxied)
    S->>T: SOCKS5 CONNECT 127.0.0.1:P
    T->>C: delivered on the server loopback
    C->>PG: vector + graph lookup (internal docker net)
    PG-->>C: rows
    C-->>O: grounded answer (back through the tunnel)

Because SOCKS resolves the destination on the server side, 127.0.0.1:P means the server's loopback — so any new container port is reachable with no extra setup.

5. Resilience & startup ordering

flowchart LR
    key["SSH key auth — add your pubkey to authorized_keys"] --> tun
    boot["host boot / app launch"] --> tun["autossh / systemd tunnel unit"]
    tun -->|"must be UP first"| orch["Orchestrator starts"]
    tun -->|"connection drops"| reconnect["auto-reconnect"]
    reconnect --> tun
    orch -.->|"all Docker + Postgres + container calls flow through"| tun

The tunnel becomes critical infrastructure: run it under autossh or a systemd unit with auto-restart, started before the Orchestrator; use SSH key auth (not the password) for an unattended tunnel.
