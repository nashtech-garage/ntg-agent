LightRAG over an SSH tunnel — architecture

The SSH-tunnel connectivity option for relocating LightRAG and its Postgres to the Azure Ubuntu VM, for the case where we cannot open an Azure NSG port but already have working SSH access.

The Orchestrator keeps driving the remote Docker daemon (Option 2), but all three runtime channels ride over one persistent SSH connection to the VM's already-open port 22 — no new inbound ports, no manager/Azure dependency.

ChannelSSH mechanismWhy
Docker daemon-L localhost:2375 → /var/run/docker.sockDrive the remote daemon; socket never exposed on TCP
Postgres (reset path)-L localhost:5432 → 127.0.0.1:5432ResetVectorSchemaAsync connects directly (only on an embedding-dimension change)
Reserved container HTTP ports-D 1080 (SOCKS5)Reach any agent's reserved lightrag-agent port in the pool with one proxy, no per-port forward

One command opens all three:

ssh -D 1080 \
    -L 2375:/var/run/docker.sock \
    -L 5432:127.0.0.1:5432 \
    ntgagent@4.193.109.6

Port model. Each agent permanently owns one host port from a configured pool (PortRangeStart…PortRangeEnd, default 20000–20999). PortReservationService assigns the lowest free port and never hands it to a second agent, so a reachable reserved port is provably that agent's own container. The container binds that exact port — ports are no longer auto-assigned by Docker and discovered by inspect.

The agent containers reach Postgres internally over the Docker network alias lightrag-postgres — that traffic never touches the tunnel. Reserved ports and Postgres bind to the VM's loopback (PortBindHostIp=127.0.0.1); the Docker daemon stays on its unix socket (never on TCP). Every agent is isolated by a LightRAG WORKSPACE=w{agentId} scoping all its rows in the shared Postgres.

2. Container spawn (Orchestrator drives the remote daemon)

Provisioning is centralised in LightRagProvisioner.ProvisionAsync: reserve the agent's port → ensure the container runs on it → on a host-port conflict (an external process holds the reserved port), reassign a new free port and retry once.

3. RAG query (reaching a reserved container port via SOCKS)

sequenceDiagram
    autonumber
    participant O as Orchestrator
    participant S as SOCKS proxy localhost:1080
    participant T as ssh -D tunnel
    participant C as lightrag-agent 127.0.0.1:P (reserved)
    participant PG as lightrag-postgres
    O->>S: GET http://127.0.0.1:P/query (proxied)
    S->>T: SOCKS5 CONNECT 127.0.0.1:P
    T->>C: delivered on the server loopback
    C->>PG: vector + graph lookup (internal docker net)
    PG-->>C: rows
    C-->>O: grounded answer (back through the tunnel)

Because SOCKS resolves the destination on the server side, 127.0.0.1:P means the server's loopback. The port P is the agent's reserved port, so a response on it is provably that agent's own container — the client factory's cached-port fast path relies on this to avoid cross-agent misrouting.

4. Reaching the reserved-port pool: plain -L vs SOCKS -D

Ports are now a bounded reserved range (20000–20999), one per agent — not random runtime ports. So plain -L forwarding could pre-forward the whole range up front. The implementation instead routes the LightRAG HttpClient through a single SOCKS -D 1080 proxy (SocksProxy, wired as a WebProxy on the client's SocketsHttpHandler).

With a bounded pool, plain -L is viable but needs one forward per reservable port. The single SOCKS -D proxy reaches every reserved port in the range — including a port freshly assigned by ReassignAsync — with no forward config to grow alongside the pool. That is the connectivity the code ships.

5. Resilience & startup ordering

flowchart LR
    key["SSH key auth — add your pubkey to authorized_keys"] --> tun
    boot["host boot / app launch"] --> tun["autossh / systemd tunnel unit"]
    orch["Orchestrator starts"] --> probe["reconciler probes daemon (Ping)"]
    probe -->|"tunnel not up yet"| probe
    probe -->|"reachable within DaemonProbeTimeout"| ready["provision containers"]
    tun -.->|"all Docker + Postgres + container calls flow through"| orch
    tun -->|"connection drops"| reconnect["auto-reconnect"]
    reconnect --> tun

Startup ordering is no longer strict. On boot the reconciler pings the daemon and retries for up to DaemonProbeTimeoutSeconds (default 60s, every DaemonProbePollIntervalMs), so the Orchestrator may start before the tunnel is up. A daemon that stays unreachable surfaces as a typed LightRagDaemonUnavailableException rather than a raw socket error. The tunnel is still critical infrastructure: run it under autossh or a systemd unit with auto-restart, and use SSH key auth (not the password) for an unattended tunnel.

6. Container readiness & idle lifecycle

flowchart LR
    start["container started on reserved port"] --> gate["readiness gate: poll GET /health"]
    gate -->|"healthy within ReadinessTimeout"| serve["endpoint returned to caller"]
    gate -->|"budget exhausted"| notready["LightRagContainerNotReadyException"]
    serve --> use["chat / upload traffic"]
    use -->|"no request for IdleTimeoutMinutes"| stop["idle-shutdown stops container (RAM reclaimed)"]
    stop -->|"next access: health miss"| reprov["factory re-provisions on the reserved port"]
    reprov --> gate

The published host port binds the instant Docker starts the container, but the LightRAG ASGI app is still booting. WaitUntilReadyAsync polls GET /health through the proxied client (a raw TCP connect cannot traverse SOCKS) every ReadinessPollIntervalMs until it answers, up to ReadinessTimeoutSeconds. Idle containers are stopped by LightRagContainerIdleShutdownService after IdleTimeoutMinutes; the next request misses the health fast path and LightRagClientFactory transparently re-provisions the container on its same reserved port.
